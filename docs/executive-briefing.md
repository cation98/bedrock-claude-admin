# Part 1: 핵심 개념 정리 (자습용)

> **한 줄 요약**: 이 플랫폼에서 사용하는 AWS/Kubernetes 핵심 기술을 호텔 비유로 쉽게 이해합니다.

이 섹션은 클라우드 인프라에 익숙하지 않은 개발자를 위해, 각 기술 개념을 일상적인 비유와 함께 설명합니다. 실제 프로젝트에서 어떻게 쓰이는지도 함께 표기합니다.

---

### 1.1 Kubernetes (K8s) — 컨테이너 오케스트레이션

**비유**: 대형 호텔의 총지배인

호텔(서버)에 수많은 투숙객(애플리케이션)이 들어오면, 총지배인은 빈 방을 찾아 배정하고, 체크아웃하면 방을 정리하고, 손님이 너무 많으면 옆 건물(추가 서버)을 열어줍니다.

Kubernetes가 하는 일도 같습니다:
- **컨테이너 배치**: 어떤 서버에서 어떤 애플리케이션을 실행할지 결정
- **자동 복구**: 애플리케이션이 죽으면 자동으로 다시 시작
- **확장/축소**: 사용자가 많아지면 자동으로 서버를 추가하고, 줄어들면 반납

**우리 프로젝트에서**: 15~1,000명의 사용자 각각에게 독립적인 Claude Code 환경(Pod)을 만들고 관리합니다.

---

### 1.2 EKS (Elastic Kubernetes Service) — AWS 관리형 Kubernetes

**비유**: 호텔 관리 위탁 서비스

Kubernetes를 직접 운영하려면 총지배인(Control Plane)을 직접 고용하고, 교육하고, 관리해야 합니다. EKS는 AWS가 이 총지배인을 대신 운영해주는 서비스입니다. 우리는 "방을 몇 개 준비해달라"고만 요청하면 됩니다.

| 구분 | 직접 운영 | EKS 사용 |
|------|----------|---------|
| Control Plane (API 서버, 데이터 저장소) | 우리가 설치/관리 | AWS가 관리 |
| Worker Node (실제 서버) | 우리가 관리 | Managed Node Group으로 AWS가 반자동 관리 |
| 비용 | 서버 비용 + 인력 비용 | 서버 비용 + EKS 비용 ($0.10/시간) |

**우리 프로젝트에서**: `bedrock-claude-eks` 클러스터를 EKS로 운영합니다. Control Plane은 AWS가 관리하고, Worker Node는 Managed Node Group으로 자동 관리됩니다.

---

### 1.3 Pod — 최소 실행 단위

**비유**: 호텔의 객실 1개

하나의 Pod = 한 사용자의 독립적인 작업 환경입니다. 각 객실(Pod)에는 다음이 갖추어져 있습니다:

| Pod 내부 구성 요소 | 역할 |
|-------------------|------|
| Claude Code CLI | AI 코딩 도우미 |
| ttyd | 웹 브라우저에서 접속하는 터미널 |
| psql | 데이터베이스(safety-prod) 조회 도구 |
| AWS CLI | AWS 서비스 호출 도구 |
| git, vim, Python | 기본 개발 도구 |

각 Pod은 서로 완전히 격리되어 있어서, A 사용자가 B 사용자의 환경에 접근할 수 없습니다.

**우리 프로젝트에서**: 사용자가 로그인하면 `claude-sessions` 네임스페이스에 Pod이 생성되고, 로그아웃하거나 4시간이 지나면 자동 삭제됩니다.

---

### 1.4 Node (EC2) — 물리적 서버

**비유**: 호텔 건물 자체

Pod(객실)이 실제로 돌아가는 물리적 서버입니다. 하나의 Node(건물)에 여러 개의 Pod(객실)이 들어갈 수 있습니다.

**우리 프로젝트에서**:
- 인스턴스 타입: `m5.large` (2 vCPU, 8GB RAM)
- Pod당 자원 제한: CPU 0.5 core, Memory 512MB
- 따라서 Node 1대당 약 **8개의 Pod**을 수용 가능 (시스템 예약 자원 제외 시)
- 리전: `ap-northeast-2` (서울)

---

### 1.5 Node Group — 서버 그룹 (Auto Scaling)

**비유**: 호텔 체인의 지점 관리

수요에 따라 건물(Node)을 자동으로 늘리거나 줄이는 그룹입니다. "최소 2동, 최대 4동"처럼 범위를 정해놓으면, 투숙객(Pod)이 늘어날 때 자동으로 건물을 추가합니다.

**우리 프로젝트에서**:

| 설정 | 값 |
|------|-----|
| 인스턴스 타입 | m5.large |
| 최소 Node 수 | 2대 |
| 희망 Node 수 | 2대 |
| 최대 Node 수 | 4대 |

---

### 1.6 ECR (Elastic Container Registry) — Docker 이미지 저장소

**비유**: 호텔 객실의 표준 인테리어 설계도 보관소

Pod을 만들 때 "어떤 소프트웨어를 설치할지"를 담은 Docker 이미지가 필요합니다. ECR은 이 이미지를 안전하게 저장하고, Node가 Pod을 만들 때 이미지를 내려받는 곳입니다.

**우리 프로젝트에서**: `claude-code-terminal` 이미지가 ECR에 저장되어 있습니다. Node.js 22 기반으로 Claude Code, ttyd, psql 등이 미리 설치된 이미지입니다.

---

### 1.7 IAM Role — 권한 인증서

**비유**: 호텔 직원 ID 카드

"이 직원은 프론트데스크만 접근 가능", "이 직원은 모든 층에 접근 가능"처럼, AWS 서비스나 리소스에 누가 무엇을 할 수 있는지를 정의합니다.

**우리 프로젝트에서 사용하는 IAM Role**:

| Role | 권한 | 용도 |
|------|------|------|
| `bedrock-claude-eks-cluster-role` | EKS 클러스터 관리 | EKS Control Plane이 AWS 리소스 관리 |
| `bedrock-claude-eks-node-role` | EC2 + ECR 읽기 | Worker Node가 클러스터에 참여하고 이미지 다운로드 |
| `bedrock-claude-bedrock-access` | Bedrock 모델 호출 | Pod 내에서 Claude API 호출 |

---

### 1.8 IRSA (IAM Roles for Service Accounts) — Pod 수준 권한 자동 주입

**비유**: 호텔 객실 카드키에 수영장 이용권이 자동 포함되는 시스템

일반적으로 AWS 접근 키를 직접 Pod에 넣어야 하지만(보안 위험), IRSA를 사용하면 Kubernetes ServiceAccount와 IAM Role을 연결하여 **Pod이 시작될 때 자동으로 임시 자격증명이 주입**됩니다.

```
흐름:
  K8s ServiceAccount (claude-terminal-sa)
    --> 연결 --> IAM Role (bedrock-access-role)
      --> 권한 --> Bedrock InvokeModel API

Pod 시작 시 AWS SDK가 자동으로 임시 자격증명을 받아옴.
별도의 Access Key/Secret Key 관리 불필요.
```

**우리 프로젝트에서**: `claude-sessions` 네임스페이스의 `claude-terminal-sa` ServiceAccount가 Bedrock 호출 권한을 가진 IAM Role과 연결되어 있습니다.

---

### 1.9 VPC / Subnet — 네트워크 격리

**비유**: 호텔 단지와 건물 내 층(Floor) 구분

- **VPC (Virtual Private Cloud)**: 호텔 단지 전체. 외부와 격리된 우리만의 네트워크 공간입니다.
- **Subnet (서브넷)**: 단지 안의 각 건물/층. Public(외부 접근 가능)과 Private(내부 전용)으로 나뉩니다.

**우리 프로젝트의 네트워크 구성**:

```
SKO VPC (10.0.0.0/16)  -- 기존 인프라, 변경하지 않음
  |
  +-- sko-public-subnet-a  (10.0.0.0/24)  -- 외부 접근 가능
  +-- sko-public-subnet-b  (10.0.4.0/24)  -- 외부 접근 가능
  +-- sko-public-subnet-c  (10.0.2.0/24)  -- 외부 접근 가능
  +-- sko-private-subnet-b (10.0.1.0/24)  -- 내부 전용 (기존)
  |
  +-- [신규] eks-private-2a (10.0.10.0/24) -- EKS 워커 노드 전용
  +-- [신규] eks-private-2c (10.0.20.0/24) -- EKS 워커 노드 전용
```

EKS 워커 노드는 Private Subnet에 배치되어 외부에서 직접 접근할 수 없습니다.

---

### 1.10 NAT Instance — 내부에서 외부로 나가는 문

**비유**: 호텔 후문 (직원 전용 출구)

Private Subnet의 리소스는 외부 인터넷에 직접 접근할 수 없습니다. NAT Instance는 "내부 -> 외부" 방향의 통신만 허용하는 게이트웨이 역할을 합니다. 외부에서 안으로 들어오는 것은 차단됩니다.

**우리 프로젝트에서**: 기존 `sko-nat-instance` (t2.small, `i-07caaab6581d70009`)를 그대로 사용합니다. EKS 전용 Private Subnet의 Route Table에 이미 연결되어 있습니다. 추가 비용이 발생하지 않습니다.

---

### 1.11 VPC Endpoint — AWS 내부 직통 통로

**비유**: 호텔과 쇼핑몰을 연결하는 지하 통로

일반적으로 AWS 서비스(Bedrock 등)를 호출하려면 인터넷을 통해 나갔다가 다시 들어와야 합니다. VPC Endpoint를 사용하면 인터넷을 거치지 않고 AWS 내부 네트워크로 직접 연결됩니다. 더 빠르고, 더 안전하고, 데이터 전송 비용도 절약됩니다.

**우리 프로젝트에서**: 기존 Bedrock VPC Endpoint (`vpce-0fab76c138da3a84a`)가 이미 구성되어 있습니다. Pod에서 Bedrock API를 호출할 때 인터넷을 거치지 않고 AWS 내부 네트워크를 통해 직접 통신합니다.

---

### 1.12 Bedrock — AWS 관리형 AI 모델 서비스

**비유**: 전기/수도처럼 쓰는 만큼만 내는 AI 서비스

직접 GPU 서버를 구매하고 모델을 설치하는 대신, AWS가 제공하는 AI 모델을 API로 호출하여 사용합니다. 사용한 토큰(글자 수) 만큼만 비용을 지불합니다.

| 항목 | 직접 호스팅 | Bedrock 사용 |
|------|-----------|-------------|
| GPU 서버 구매 | 필요 (수천만원~) | 불필요 |
| 모델 설치/업데이트 | 직접 관리 | AWS가 자동 관리 |
| 가용성/장애 대응 | 직접 운영 | AWS SLA 보장 |
| 비용 구조 | 고정비 (사용 안 해도 지불) | 종량제 (사용한 만큼만) |
| 데이터 보안 | 직접 관리 | AWS 계정 내 격리, 모델 학습에 사용 안 됨 |

**우리 프로젝트에서 사용하는 모델**:
- **Claude Sonnet 4.6** (`us.anthropic.claude-sonnet-4-6`): 주력 모델 - 코딩, 분석, 문서 생성
- **Claude Haiku 4.5** (`us.anthropic.claude-haiku-4-5`): 경량 모델 - 단순 Q&A, 비용 절감용

# Part 2: 시스템 아키텍처

> **한 줄 요약**: 사용자가 SSO 로그인하면 개인 전용 AI 코딩 환경(Pod)이 자동 생성되고, 웹 브라우저에서 바로 Claude Code를 사용할 수 있습니다.

---

### 2.1 전체 흐름 (사용자 관점)

```
[사용자]
    |
    | 1) 웹 브라우저에서 claude.skons.net 접속
    v
[SSO 로그인] (sso.skons.net)
    |
    | 2) 사번/비밀번호로 인증
    v
[Auth Gateway] (FastAPI)
    |
    | 3) 인증 성공 -> JWT 토큰 발급
    | 4) Kubernetes API 호출 -> 개인 Pod 생성
    v
[EKS Cluster]
    |
    | 5) Pod 생성 (Claude Code + 웹 터미널 + DB 접근 도구)
    | 6) AWS 임시 자격증명 자동 주입 (IRSA)
    v
[사용자 Pod]
    |
    +---> [Bedrock API] : Claude에게 질문/코딩 요청
    +---> [RDS Replica] : safety-prod DB 조회 (읽기 전용)
    |
    | 7) 웹 터미널 URL 반환
    v
[사용자 웹 브라우저]
    브라우저에서 터미널 화면이 열리고,
    Claude Code를 바로 사용 가능
```

---

### 2.2 주요 구성 요소

| 구성 요소 | 기술 스택 | 역할 |
|-----------|----------|------|
| Auth Gateway | FastAPI (Python 3.12) | SSO 인증, JWT 발급, Pod 생성/삭제 관리 |
| Admin Dashboard | Next.js 15 + React | 세션 모니터링, 사용량 조회, 사용자 관리 |
| Container Image | Docker (Node.js 22 기반) | 사용자별 독립 실습 환경 (Claude Code + ttyd + psql) |
| EKS Cluster | Kubernetes 1.31 | Pod 오케스트레이션, 자동 확장 |
| Infra as Code | Terraform | AWS 리소스 선언적 관리 |

---

### 2.3 보안 설계

| 보안 영역 | 적용 방식 |
|-----------|----------|
| 네트워크 격리 | EKS Worker Node는 Private Subnet에서만 운영 |
| Pod 간 격리 | NetworkPolicy로 Pod 간 통신 완전 차단 |
| 자격증명 | IRSA로 임시 자격증명 자동 주입 (1~4시간 유효) |
| DB 접근 | ReadOnly Replica만 연결 (데이터 변조 불가) |
| 세션 관리 | 최대 TTL 4시간, 초과 시 Pod 자동 삭제 |
| 데이터 보안 | 모든 데이터는 SKO AWS 계정 내에서만 처리, Bedrock 모델 학습에 사용되지 않음 |

# Part 3: 비용 분석

> **한 줄 요약**: Phase 1(15명, 1주)은 약 $100, 전사 확대(1,000명)는 월 약 $14,300~$16,000이며, 최적화 적용 시 월 $10,000 이하로 절감 가능합니다.

이 섹션은 임원 보고용으로, 표를 그대로 프레젠테이션에 활용할 수 있도록 구성했습니다.

---

### 3.1 인프라 비용 (고정비 — 사용하지 않아도 발생)

| 리소스 | 단가 | Phase 1 (15명) | Phase 2 (50명) | 전사 (1,000명) |
|--------|------|---------------|---------------|---------------|
| EKS Cluster | $0.10/시간 | $73/월 | $73/월 | $73/월 |
| EC2 Node (m5.large) | $0.096/시간 (서울 리전) | 2대 = $140/월 | 7대 = $490/월 | 20대 = $1,400/월 (피크) |
| ECR 저장소 | $0.10/GB/월 | ~$0.10 | ~$0.10 | ~$0.10 |
| NAT Instance | 기존 t2.small 사용 | $0 (추가비용 없음) | $0 | $0 |
| Bedrock VPC Endpoint | 기존 사용 | $0 (추가비용 없음) | $0 | $0 |
| **인프라 소계** | | **~$213/월** | **~$563/월** | **~$1,473/월** |

> **참고**: m5.large 1대 = 2 vCPU, 8GB RAM. Pod당 CPU 0.5 core + Memory 512MB 할당 기준, Node 1대당 약 8명 수용 가능.

---

### 3.2 Bedrock 사용량 비용 (변동비 — 사용한 만큼만 발생)

#### 모델별 단가

| 모델 | Input (입력) | Output (출력) | 용도 |
|------|-------------|-------------|------|
| Claude Sonnet 4.6 | $3 / 1M 토큰 | $15 / 1M 토큰 | 코딩, 분석, 문서 생성 |

# Part 4: 1,000명 규모 아키텍처 변경 사항

> **한 줄 요약**: 1,000명 규모에서는 자동 확장, 모델 라우팅, 사전 준비된 Pod 풀, 부서별 사용량 제한 등의 아키텍처 개선이 필요합니다.

현재 Phase 1 아키텍처(15명)에서 전사 규모(1,000명)로 확장할 때 변경/추가해야 하는 사항입니다.

---

### 4.1 Cluster Autoscaler / Karpenter (자동 노드 확장)

**현재**: Node Group의 min/max를 수동으로 설정 (min=2, max=4)

**변경**: Karpenter 또는 Cluster Autoscaler를 도입하여 Pod 수요에 따라 Node를 자동으로 추가/제거

```
현재 방식:
  관리자가 "최대 4대"로 미리 설정 -> Pod이 더 필요하면 수동 변경

Karpenter 도입 후:
  Pod이 스케줄링 안 됨 -> Karpenter가 자동으로 최적 EC2 생성 (30초 이내)
  Pod이 줄어듦 -> 사용하지 않는 Node 자동 삭제 (비용 절감)
```

**권장 사항**: Karpenter는 AWS가 직접 개발한 오픈소스로, Cluster Autoscaler보다 빠르고 유연합니다. Spot Instance와의 연동도 더 우수합니다.

---

### 4.2 모델 라우팅 (Intelligent Model Routing)

**현재**: 모든 사용자가 Sonnet 4.6 (고비용 모델) 사용

**변경**: 요청 유형에 따라 적절한 모델로 자동 라우팅

| 요청 유형 | 라우팅 모델 | 단가 비교 | 예시 |
|-----------|-----------|----------|------|
| 복잡한 코딩/분석 | Sonnet 4.6 | $3/$15 | 아키텍처 설계, 코드 리팩토링, 복잡한 디버깅 |
| 단순 Q&A/DB 조회 | Haiku 4.5 | $0.80/$4 | "이 테이블 구조 알려줘", 간단한 코드 수정 |

**구현 방식**: Auth Gateway에 모델 라우팅 로직 추가. 사용자 역할(사무실/현장) 또는 요청 복잡도에 따라 자동 분기.

---

### 4.3 Shared Terminal Pool (사전 준비된 Pod 풀)

**현재**: 사용자 로그인 시 Pod 생성 (약 30초~1분 대기)

**변경**: 미리 생성해둔 Pod 풀에서 즉시 할당

```
현재:
  로그인 -> Pod 생성 요청 -> 이미지 다운로드 -> 컨테이너 시작 -> 30~60초 대기

Pod Pool 도입 후:
  미리 10~20개 Pod을 "대기 상태"로 준비
  로그인 -> 대기 Pod 즉시 할당 -> 3초 이내 접속
  할당된 Pod 수만큼 새로운 대기 Pod 자동 보충
```

**효과**: 사용자 경험 대폭 개선 (1분 대기 -> 3초 이내), 임원 시연 시 즉각적인 반응.

---

### 4.4 API Gateway 계층 (Rate Limiting / 사용량 제한)

**현재**: 사용량 제한 없음

**변경**: 부서별/개인별 사용량 제한 (Quota) 도입

| 제한 유형 | 적용 대상 | 제한 값 (예시) |
|-----------|----------|--------------|
| 일일 토큰 한도 | 개인 | 사무실 300K / 현장 50K |
| 분당 요청 수 (RPM) | 개인 | 20 RPM |
| 월간 부서 예산 | 부서 | 부서별 $1,000~$5,000 |

**효과**: 비용 예측 가능성 확보, 특정 사용자의 과도한 사용 방지, 부서별 비용 배분.

---

### 4.5 아키텍처 변경 요약표

| 항목 | Phase 1 (15명) | 전사 (1,000명) | 변경 이유 |

### 5.2 권장 방안: 경량 채팅 인터페이스 + Claude Skills

#### Claude Skills란?

사전에 정의된 프롬프트 템플릿으로, 사용자가 복잡한 명령을 몰라도 "DB 조회", "보고서 생성" 같은 버튼을 누르면 자동으로 최적화된 프롬프트가 실행됩니다.

```
기존 방식 (풀 터미널):
  사용자: "safety-prod DB에서 최근 1주일 안전 점검 결과를 조회해서
           부서별로 정리해줘. 미통과 항목은 별도로 표시해줘."
  -> Claude가 전체 맥락 이해 -> SQL 생성 -> 실행 -> 결과 정리
  -> 토큰 사용: ~50K (매번 전체 맥락 전송)

Claude Skills 방식 (템플릿):
  사용자: [안전 점검 조회] 버튼 클릭 -> 기간: 최근 1주 선택
  -> 사전 정의된 프롬프트 + SQL 템플릿 자동 실행
  -> 토큰 사용: ~5K (템플릿이므로 맥락 최소화)
```

#### 비용 비교

| 항목 | 풀 터미널 (Sonnet) | 경량 채팅 (Haiku + Skills) | 절감률 |
|------|-------------------|--------------------------|--------|
| 모델 단가 (Input) | $3 / 1M 토큰 | $0.80 / 1M 토큰 | 73% 절감 |
| 모델 단가 (Output) | $15 / 1M 토큰 | $4 / 1M 토큰 | 73% 절감 |
| 1인 일일 토큰 | ~30K | ~8K (Skills 최적화) | 73% 절감 |
| 1인 일일 비용 | ~$0.25 | ~$0.05 | 80% 절감 |
| 700명 월 비용 | **~$3,850** | **~$770** | **~$3,080/월 절감** |

#### 인프라 절감

경량 채팅 인터페이스는 Pod당 자원 사용량이 훨씬 적습니다:

| 항목 | 풀 터미널 Pod | 경량 채팅 |
|------|-------------|----------|
| CPU | 0.5 core | 불필요 (서버사이드 처리) |
| Memory | 512MB | 불필요 |
| 형태 | 사용자당 1 Pod | 공유 웹 서버 1~2대 |
| EC2 비용 | 700명 / 8 = 88대 필요 (이론상) | 웹 서버 2~3대 |

> 실제로는 동시 접속률이 낮아 88대까지는 불필요하지만, 풀 터미널 대비 인프라 비용이 크게 줄어드는 것은 확실합니다.

| 항목 | 현재 상태 | 설명 |
|------|---------|------|
| 데이터 격리 | **안전** | 모든 데이터는 SKO AWS 계정 내에서만 처리됨. 외부 전송 없음 |
| 모델 학습 차단 | **안전** | Bedrock은 고객 데이터를 모델 학습에 사용하지 않음 (AWS 공식 정책) |
| DB 접근 권한 | **안전** | ReadOnly Replica만 연결. 원본 데이터 변조 불가 |
| 민감 정보 필터링 | **Phase 2+ 계획** | Bedrock Guardrails 기능으로 민감 데이터(개인정보, 영업비밀 등) 자동 필터링 예정 |
| 네트워크 격리 | **안전** | EKS Worker Node는 Private Subnet에서만 운영. 외부 직접 접근 차단 |
| 감사 로그 | **기본 구현** | 세션 시작/종료, 사용량 기록. 전사 확대 시 CloudTrail + CloudWatch 연동 강화 필요 |

---

### 6.3 운영 리스크

| 리스크 | 대응 방안 |
|--------|----------|
| 비용 초과 | 부서별/개인별 사용량 Quota 설정, 월간 예산 알림 구성 |
| 사용자 교육 부족 | Phase 1(임원 15명)에서 교육 콘텐츠 검증 후 단계별 확대 |
| 장애 시 업무 중단 | Claude Code는 보조 도구이므로 기존 업무 프로세스는 유지. 의존도 관리 필요 |
| 불필요한 사용 증가 | 사용량 모니터링 대시보드 + 관리자 알림으로 이상 사용 탐지 |

### 6.4 AWS 서비스 Quota 참고

Bedrock Claude 모델의 기본 Quota (리전별, 계정별):

| Quota 항목 | 기본값 (예시) | 1,000명 필요 추정치 | 조치 |
|-----------|-------------|-------------------|------|
| RPM (분당 요청 수) | ~100 RPM | ~500 RPM | AWS Support에 할당량 증대 요청 |
| TPM (분당 토큰 수) | ~100K TPM | ~1M TPM | AWS Support에 할당량 증대 요청 |

> **중요**: Quota 증대 요청은 승인까지 1~5영업일이 소요될 수 있으므로, 전사 확대 최소 2주 전에 요청해야 합니다.

## 부록 A: 용어 사전

| 용어 | 의미 |
|------|------|
| Pod | Kubernetes의 최소 실행 단위. 1 Pod = 1 사용자 환경 |
| Node | Pod이 실행되는 물리 서버 (EC2 인스턴스) |
| EKS | AWS가 관리해주는 Kubernetes 서비스 |
| ECR | Docker 이미지 저장소 (Container Registry) |
| IAM | AWS 접근 권한 관리 시스템 |
| IRSA | Pod에 AWS 자격증명을 자동 주입하는 기능 |
| VPC | 가상 사설 네트워크 (격리된 네트워크 공간) |
| Subnet | VPC 내부의 네트워크 구간 (Public/Private) |
| NAT | Private 리소스의 외부 인터넷 접근을 중계 |
| VPC Endpoint | AWS 내부 네트워크로 직접 서비스 연결 (인터넷 불필요) |
| Bedrock | AWS의 관리형 AI 모델 서비스 (종량제) |
| Sonnet / Haiku | Anthropic Claude 모델의 등급 (성능/비용 차이) |
| Token | AI 모델이 처리하는 텍스트 단위 (~4글자 = 1토큰) |
| RPM | Requests Per Minute (분당 요청 수) |
| TPM | Tokens Per Minute (분당 토큰 수) |
| TTL | Time To Live (자동 만료 시간) |
| STS | Security Token Service (임시 자격증명 발급) |
| OIDC | OpenID Connect (인증 프로토콜) |
| Terraform | 인프라를 코드로 관리하는 도구 (IaC) |

## 부록 B: 의사결정 요약 (임원 보고용 1-Pager)

### 프로젝트: 사내 AI 코딩 플랫폼 (Bedrock Claude Code)

**목적**: 임직원에게 AI 코딩 도구(Claude Code)를 안전한 사내 환경에서 제공

**핵심 장점**:
- 데이터가 회사 AWS 계정 밖으로 나가지 않음 (보안)
- 사용한 만큼만 비용 지불 (종량제)
- 단계적 확대 가능 (15명 -> 50명 -> 1,000명)

**비용 요약**:

| 단계 | 대상 | 비용 | 기간 |
|------|------|------|------|
| Phase 1 | 임원 15명 실습 | ~$100 | 1주 |
| Phase 2 | 팀장 50명 실습 | ~$300 | 2주 |
| 전사 배포 | 1,000명 | ~$15,000/월 (최적화 전) | 상시 |
| 전사 배포 (최적화) | 1,000명 | **~$8,800/월 (약 1,230만원)** | 상시 |

**주요 최적화 수단**:
1. 현장 근무자(700명)는 경량 채팅 + 저비용 모델 사용 -> $3,850 -> $770/월
2. 업무 시간 외 인프라 자동 축소 -> EC2 비용 60% 절감
3. 반복 작업 템플릿화 -> 토큰 비용 50~70% 절감

**필요 조치**:
- Phase 1 즉시 시작 가능 (인프라 준비 완료)
- 전사 확대 시 AWS Bedrock Quota 증대 요청 필요 (2주 사전 신청)
- 현장 근무자용 경량 인터페이스 별도 개발 필요 (Phase 2 이후)
